## Project Overview

**Moments** is a fully browser-based photo & video slideshow editor. Users import media, arrange clips on a timeline, apply transitions and text overlays, and export a finished MP4 — all without a server. Video encoding runs inside the browser via FFmpeg compiled to WebAssembly.

**Stack**

| Layer | Tool |
|---|---|
| Framework | React 19 + Vite 8 |
| Video encoding | `@ffmpeg/ffmpeg` 0.12 (WebAssembly / FFmpeg 5.1) |
| Icons | `lucide-react` |
| Styling | CSS Modules |
| State | Custom hook (`useMediaStore`) — no external state lib |

**Run locally**
```bash
npm install
npm run dev   # http://localhost:5173
```

> Vite's dev server must emit `Cross-Origin-Opener-Policy` and `Cross-Origin-Embedder-Policy` headers for `SharedArrayBuffer` (required by FFmpeg.wasm). This is already configured in `vite.config.js`.

## ⚠️ WASM Memory — CRITICAL

The FFmpeg WASM binary (`public/ffmpeg/esm/ffmpeg-core.wasm` and `public/ffmpeg/ffmpeg-core.wasm`) has been **patched** to increase its initial heap from 32 MB (512 pages) to **128 MB (2048 pages)**.

### Why this patch exists

The FFmpeg WASM heap starts at 32 MB and grows on demand. In browsers, the `drawtext` filter at 720p/1080p requires a sudden ~36 MB allocation for decode frame buffers + FreeType font cache + libx264 encode buffers simultaneously. Browsers refuse large one-shot heap growth requests even when total memory is available, causing `Aborted()` in Stage 3. The 128 MB pre-allocation reserves contiguous virtual address space upfront, before the browser maps other tab resources, preventing this failure.

### Patch details

The patch is a 2-byte change at offset **25620** in both WASM files:

| Value | Pages | MB | Bytes at offset 25620 |
|---|---|---|---|
| Before (original) | 512 | 32 MB | `0x80 0x04` |
| After (patched) | 2048 | 128 MB | `0x80 0x10` |

The WASM max (32768 pages = 2 GB) and file sizes are unchanged.

### Re-applying the patch

If the WASM files are ever replaced with a fresh download from npm, **re-apply the patch** before shipping:

```js
// patch-wasm.js — run with node
const fs = require('fs')
for (const path of [
  'public/ffmpeg/esm/ffmpeg-core.wasm',
  'public/ffmpeg/ffmpeg-core.wasm',
]) {
  const buf = Buffer.from(fs.readFileSync(path))
  if (buf[25620] === 0x80 && buf[25621] === 0x04) {
    buf[25621] = 0x10  // 512 pages → 2048 pages (32MB → 128MB)
    fs.writeFileSync(path, buf)
    console.log('Patched:', path)
  } else if (buf[25620] === 0x80 && buf[25621] === 0x10) {
    console.log('Already patched:', path)
  } else {
    console.error('Unexpected bytes — patch offset may have changed:', path, buf[25620].toString(16), buf[25621].toString(16))
  }
}
```

## ⚠️ Export Pipeline — DO NOT BREAK

**The export pipeline is extremely sensitive to changes.** Any modification to `useFFmpeg.js` must preserve all of the following invariants or the export will crash with `Aborted()` or produce corrupt output.

### Threading — MUST NOT CHANGE

All FFmpeg commands use **`-threads 1`** hardcoded everywhere in the pipeline. This is not a performance setting — it is a stability requirement for FFmpeg.wasm's WASM execution environment.

- **Do not add an encoder thread count setting** (previously called "Export Speed"). Multi-threaded libx264 encodes fragment the WASM heap across all pipeline stages. By Stage 3, the accumulated fragmentation prevents the drawtext filter from allocating its working memory at 720p+, causing `Aborted()`. This crash is environment-specific and will not reproduce in Node.js testing.
- `-filter_threads 1` in Stage 3 is also mandatory and must not be changed or removed.

### WASM FS is always fresh

The FFmpeg WASM instance is **terminated and recreated after every export** (success or failure). The WASM FS starts empty on every new instance. There is no pre-export file cleanup loop — it was removed as it was always a no-op (768 delete calls on an empty FS) and added measurable startup latency. Do not re-add it.

**Exception:** `IMAGE_EXTS` must be declared inside `exportMoment()` — it was previously declared inside the cleanup block. It is still required by the source file writing loop (`_isImg: IMAGE_EXTS.has(ext)`). Do not remove it.

```js
// Required at the top of exportMoment() try block:
const IMAGE_EXTS = new Set(['jpg','jpeg','png','gif','webp','heic','avif'])
```

### ArrayBuffer transfer — font data MUST be `.slice()`'d

`ff.writeFile()` in `@ffmpeg/ffmpeg 0.12` transfers `Uint8Array` data to the Worker via `postMessage` with transfer semantics. After the transfer, the original `ArrayBuffer` is **detached** and becomes unusable. Any subsequent export that tries to write the same bytes will crash with `DataCloneError: ArrayBuffer is already detached`.

**Rules:**
- All font bytes written to the WASM FS must be fresh copies: `fontBytes.slice()`
- This applies to both preset fonts (from `fontDataCache`) and custom fonts (`seg.customFontData`)
- Custom font bytes must also be `.slice()`'d when stored into React state (in `Inspector.jsx` upload handler and in `useMediaStore.loadWorkflow`) so that the copy in state is never the transferred original

```js
// CORRECT — Stage 3 font write:
await ff.writeFile(`font_${key}.ttf`, fontBytes.slice())          // preset font
await ff.writeFile(`font_${key}.ttf`, seg.customFontData.slice()) // custom font

// WRONG — will crash on second export:
await ff.writeFile(`font_${key}.ttf`, fontBytes)
await ff.writeFile(`font_${key}.ttf`, seg.customFontData)
```

### Preset fonts must be pre-fetched at load time

Preset font files are fetched from `/ffmpeg/fonts/` **during `load()`** (before any export) and cached as `Uint8Array` in `fontDataCache` (a `useRef`). Stage 3 writes from this cache using `.slice()`.

Do not call `fetchFile()` for fonts inside the export pipeline (inside the Worker context). Under strict `COOP/COEP` headers in some browsers, Worker-initiated fetches can fail or return corrupt data at higher resolutions, causing `Aborted()`.

### `run()` helper checks return code

The `run()` helper calls `await ff.exec(args)` and checks `ret !== 0`. When FFmpeg aborts, the WASM core catches the exception internally and returns a non-zero integer rather than throwing. The `run()` helper must check this return value — a `try/catch` alone is insufficient:

```js
const run = async (label, args) => {
  let ret
  try { ret = await ff.exec(args) } catch (e) { /* ... throw */ }
  if (ret !== 0) {
    // push tail log and throw
  }
}
```

Note: `run()` no longer takes a third `outFile` argument. File existence is not verified by reading — that was removed to eliminate the double-memory-load that caused OOM crashes at 720p.

## State Architecture — `useMediaStore`

Single custom hook. `App.jsx` calls it once and passes slices down as props — no context, no global store.

### Clip object schema

```js
{
  id:               string,       // 'clip_N'
  file:             File | null,
  url:              string | null,
  name:             string,
  type:             'image' | 'video',
  duration:         number,       // Timeline slot duration (seconds)
  fileDuration:     number | null,// Full raw video duration from metadata
  trimStart:        number,       // Seek point into source (seconds)
  trimEnd:          number | null,// End cut point (null = no explicit trim)
  brightness:       number,       // -50 to +50
  contrast:         number,       // -50 to +50
  saturation:       number,       // -50 to +50
  speed:            number,       // 0.25–4 (video only)
  transition:       string | null,// Per-clip override; null = use global
  transitionDuration: number | null,
  viewZoom:         number,       // Saved pan/zoom — 1 = no zoom
  viewPanX:         number,       // Pixels (canvas-space)
  viewPanY:         number,
  includeAudio:     boolean,
  blurBackground:   boolean,
  imageEffect:      string | null,// 'ken_burns'|'pan_zoom'|'parallax'|'fade_in'|null
  mediaId:          string,
  _needsMedia:      boolean,      // True after workflow load — cleared when file is re-added by name match
}
```

### Music state

```js
musicFile:        File | null    // The audio file object
musicDuration:    number | null  // Full duration in seconds (populated from <audio> onLoadedMetadata)
musicTrimStart:   number         // Start of selected music segment (seconds, default 0)
musicTrimEnd:     number | null  // End of selected segment (null = play to end)
```

`setMusicFile(file)` is a wrapper — calling it always resets `musicDuration`, `musicTrimStart`, and `musicTrimEnd` to their defaults. Never call `setMusicFileRaw` directly.

### End Fade state

```js
endFadeVideo:         boolean  // Enable fade-to-black at end of timeline
endFadeVideoDuration: number   // Duration of video fade (0.5–5s)
endFadeAudio:         boolean  // Enable audio fade-out at end of timeline
endFadeAudioDuration: number   // Duration of audio fade (0.5–5s)
```

### Computed export values

```js
// Trim-accurate output length — used by Timeline chip and as master duration reference.
// Subtracts transition overlaps (N clips → N-1 transitions).
exportDuration = (() => {
  const rawTotal = clips.reduce((s, c) => {
    if (c.type === 'video' && c.fileDuration) {
      return s + (trimEnd - trimStart) / speed
    }
    return s + (c.duration || 0)
  }, 0)
  let transitionOverlap = 0
  for (let i = 0; i < clips.length - 1; i++) {
    const hasTransition = (clips[i].transition || globalTransition) !== 'none'
    if (hasTransition) transitionOverlap += clips[i].transitionDuration ?? transitionDuration
  }
  return Math.max(0, rawTotal - transitionOverlap)
})()
```

### Workflow save/load (version 10)

`saveWorkflow()` serialises to JSON and triggers a browser download. The current workflow version is **10**.

Saved fields include all clip settings, text segments, music trim, end fade settings, `quality`, `musicVolume`, and a `customFonts` table for any uploaded fonts.

#### Fields saved per version

- **v10+**: `quality`, `musicVolume` included. `customFonts` table keyed by content hash (`name_hash`) to avoid collisions when two segments use different fonts with the same filename. Each text segment carries `_fontKey` to look up the correct font entry.
- **v9**: `customFonts` table keyed by filename only. `quality` and `musicVolume` not saved; defaults (720p, 70) apply on load.

`loadWorkflow(json)` handles both v9 and v10 automatically. Version guard logs a warning for versions > 10 but still attempts load.

#### Custom font persistence

Custom fonts uploaded by the user are saved in a top-level `customFonts` table in the workflow JSON. On load, font bytes are decoded from Base64 and stored with `.slice()` to ensure each segment's `customFontData` owns a fresh, non-transferable `ArrayBuffer`.

```js
// Save: Uint8Array → chunked String.fromCharCode → btoa → Base64 string
// Key:  `${fontName}_${contentHash}` (v10) or fontName (v9)
// Skip: buffers with byteLength === 0 (already detached — silently omit)

// Load: atob → binary string → Uint8Array → .slice() per segment
// After load, Preview.jsx detects customFontData and registers it via FontFace API.
```

`mediaId` is **not** a stable key — it changes each session. Clips are matched on reload by **filename** (`c.name`).

`loadWorkflow(json)` sets all state and marks clips `_needsMedia: true`. When the user re-adds files via the Media Library, `addToLibrary` matches each new file to any `_needsMedia` clip with the same `name`, wires up `file`, `url`, `mediaId`, and clears `_needsMedia` automatically.

## Preview Viewport — Visual Contract

**The exported video must match what the user sees in the Preview component. This is non-negotiable.**

### How the Preview renders images

`Preview.jsx` renders images inside a `mediaLayer` div:

```jsx
<div style={{ transform: `translate(${pan.x}px, ${pan.y}px) scale(${zoom})`,
              transformOrigin: 'center center' }}>
  <img style={{ width: '100%', height: '100%', objectFit: 'contain' }} />
</div>
```

Key rules:

1. **`object-fit: contain`** — image scales to fit the W×H canvas preserving aspect ratio. Black bars (or blurred bg fill) appear on sides or top/bottom.
2. **`scale(zoom)` from centre** — zoom enlarges from canvas centre. Zoom 2× means the visible region is `W/2 × H/2`.
3. **`translate(panX, panY)`** applied before scale — pan shifts in canvas pixels. Export must apply the same transform.

### Text overlay visibility

When not playing, the Preview only shows text segments that **overlap the currently displayed clip's time window** (`clipTimeStart` to `clipTimeEnd`). This is a clip-level approximation, not exact playhead matching. When playing, segments are filtered by the live `playTime` value.

## ⚠️ FFmpeg Instance Lifecycle — CRITICAL

**The FFmpeg WASM instance is terminated and recreated after every export.**

```js
// In the exportMoment() finally block:
ffmpegRef.current.terminate()
ffmpegRef.current = null
loadedRef.current = false
setLoaded(false)
```

**Why:** FFmpeg.wasm accumulates internal libavcodec/libavfilter state between runs. Re-using the same instance for a second export causes `Aborted()` in the `drawtext` filter graph.

**Consequence:** Every export begins with `f.loaded === false`, which triggers `f.load()`. The WASM binary is browser-cached after the first load so subsequent reloads are fast (~1–2s), but the "Loading FFmpeg engine…" state will always appear briefly before each export.

**Never** attempt to reuse the FFmpeg instance across exports or skip the terminate step.

## Export Pipeline — `useFFmpeg.js` → `exportMoment()`

### Signature

```js
  onProgress,

```

All values are read from `storeRef.current` in `App.jsx` — never from intermediate refs or local state.

### Stage 1 — Per-clip encoding → `seg_N.mp4`

#### `actualDur` calculation (CRITICAL — must match `useMediaStore.exportDuration`)

```js
// Video clips:
const effectiveTrimEnd = (c.trimEnd && c.trimEnd > trimStart)
  ? c.trimEnd
  : (c.fileDuration || c.duration || 4)   // fileDuration is the correct fallback
actualDur = (effectiveTrimEnd - trimStart) / speed
actualDur = Math.max(td + 0.5, actualDur) // ensure segment is longer than transition

// Image clips:
actualDur = Math.max(td + 0.5, c.duration || 4)
```

**Never fall back to `c.duration` alone for video clips** — `c.duration` is the UI slot and may differ from `c.fileDuration`.

#### Image clip sub-process

```
Pass 1 (single pass) — PLACEMENT + optional BLUR + optional EFFECT/EQ + SILENCE + NORMALISE → seg_N.mp4
```

All image processing is collapsed into a **single FFmpeg invocation**:

- If `blurBackground`: `filter_complex` uses `split=2` — one fork generates the full-bleed boxblur background `[_bg]`, the other generates the fitted/placed foreground `[_fg]`; both are overlaid to produce `[vout]`.
- If `imageEffect` or `eqFilter`: chained directly onto `[vout]` → `[vfinal]` within the same `filter_complex`. No intermediate file is written.
- Silence audio (`lavfi anullsrc`) is muxed as a second input in the same command.
- Normalise flags (`-r 30 -fps_mode cfr -video_track_timescale 90000`) are applied as output flags on this single pass.

There are no intermediate files. No `blurbase_N.mp4`, no `placed_N.mp4`, no `effect_N.mp4`, no `aud_N.aac`.

#### Video clip sub-process

```
Pass 1 (single pass) — PLACEMENT + optional BLUR + NORMALISE → combined with audio → seg_N.mp4
Step B                — AUDIO (real audio or silence) → aud_N.aac
```

- If `blurBackground`: `filter_complex` uses `split=2` — same pattern as images. Both chains from the single source input.
- If no blur: `-vf` placement chain.
- Audio (Step B) is encoded separately as `aud_N.aac` first, then muxed into the terminal visual pass as a second `-i` input.
- Normalise flags are applied as output flags on the terminal pass. No separate normalise/mux step.
- `aud_N.aac` is deleted after the terminal pass.

### Stage 2 — Transitions, music, and music trim → `prefinal.mp4`

Stage 2 receives only the normalised `seg_N.mp4` files. xfade time offsets use `seg.actualDur` (real encoded duration from Stage 1), not `c.duration` from clip state.

#### Transition handling

Stage 2 has three paths:

**Path A — concat demuxer (stream copy, zero re-encode)**
When `multiClip && !hasMusicFile` and every transition is `'none'`, Stage 2 writes a concat playlist to the WASM FS and runs:
```
-f concat -safe 0 -i concat_list.txt -c copy prefinal.mp4
```
This is the fastest possible path for cut-only exports. All `seg_N.mp4` files have identical codec/timebase parameters so stream-copy is safe.

**Path B — single clip**
If only one segment exists, `s2-copy` stream-copies it to `prefinal.mp4`. If music is present, it is mixed via `amix`.

**Path C — xfade / mixed transitions**
Multiple clips with at least one non-`'none'` transition. All segments are loaded as separate inputs. The filter_complex chain is built dynamically:
- `transition === 'none'`: `concat=n=2` filter for hard cut. `timeOff += seg.actualDur` (no overlap).
- any other transition: `xfade` + `acrossfade`. `timeOff += seg.actualDur - td`.

**⚠️ Timebase fix for mixed transitions (CRITICAL):** The `concat` filter resets stream timebase to `1/1000000`. When a `concat` node is followed by an `xfade` node in the same filter_complex (mixed-transition export), the timebase mismatch causes `Failed to configure output pad on Parsed_xfade_N` / `Aborted()`. Fix: when a `concat` is not the last join in the chain, immediately append `settb=1/90000,setpts=PTS` (video) and `asetpts=PTS` (audio) normalisation nodes after it before passing to the next xfade.

```js
// Non-terminal concat → normalise tb before next xfade
vParts.push(`${vLabel}[${i+1}:v]concat=n=2:v=1:a=0${vTmp}`)
vParts.push(`${vTmp}settb=1/${SEG_TIMEBASE},setpts=PTS${vOut}`)
aParts.push(`${aLabel}[${i+1}:a]concat=n=2:v=0:a=1${aTmp}`)
aParts.push(`${aTmp}asetpts=PTS${aOut}`)
```

#### Music trim

If `musicTrimStart > 0` or `musicTrimEnd != null`, an `atrim` + `asetpts` filter is prepended to the music audio stream:

```js
const musicTrimFilter = hasMusicTrim
  ? `atrim=start=${musicTrimStart}${musicTrimEnd != null ? `:end=${musicTrimEnd}` : ''},asetpts=PTS-STARTPTS,`
  : ''
```

`asetpts=PTS-STARTPTS` is mandatory — without it, `amix` sees timestamps starting mid-stream and the mix misaligns.

### Stage 3 — Text overlays → `output.mp4`

All `drawtext` filters are combined into a single `-vf` pass on `prefinal.mp4`. Audio is carried through unchanged with `-c:a copy`.

#### Threading — MUST NOT CHANGE

```js
await run('s3-text', [
  '-threads', '1',         // encoder threads — must stay 1
  '-filter_threads', '1',  // filter graph threads — MUST stay 1
  '-i', preFinal,
  '-vf', vfChain,
  ...
])
```

`-filter_threads 1` is **intentional and mandatory**. FFmpeg.wasm's pthread implementation is unstable under the `drawtext` filter graph at 720p+ when multiple filter threads are active. Unlike the encoder thread count, this value must never be increased.

#### Font file handling

Font files are written to the WASM FS immediately before the Stage 3 encode. Every write uses `.slice()` to produce a fresh `Uint8Array` so the cached/state copy is never detached:

```js
await ff.writeFile(`font_${key}.ttf`, fontBytes.slice())          // preset font
await ff.writeFile(`font_${key}.ttf`, seg.customFontData.slice()) // custom font
```

After the encode completes, all font files are deleted.

Font paths use an absolute `/` prefix in the WASM FS (e.g. `/font_Poppins-Regular.ttf`) to ensure `drawtext` can locate them regardless of the WASM working directory.

#### `buildDrawtext` notes

- Uses `enable='between(t,t0,end)'` for time gating.
- Alpha expression uses a 2-level `if()` (not 4-level) to stay within FFmpeg.wasm's expression evaluator stack limit at 1080p.
- Font size is scaled by `W / 1280` so text appears visually consistent across all quality presets.

---

### Stage 4 — End fade → `output.mp4`

Applied only when `endFadeVideo` or `endFadeAudio` is true.

```js
// Video fade:  fade=t=out:st=<totalDur - fadeDur>:d=<fadeDur>:color=black
// Audio fade:  afade=t=out:st=<totalDur - fadeDur>:d=<fadeDur>
```

Fade durations are clamped to the total encoded duration. When only audio fades, `-c:v copy` is used so the video stream is not re-encoded. The output is renamed from `output_faded.mp4` to `output.mp4` via `ff.rename()`.

## Uniform Stream Specification

Every `seg_N.mp4` delivered to Stage 2 **must** conform exactly:

| Property | Value | Constant |
|---|---|---|
| Video codec | libx264 | — |
| Pixel format | yuv420p | `SEG_PIX_FMT` |
| Frame rate | 30 fps CFR | `SEG_FPS` |
| Timebase | 1/90000 | `SEG_TIMEBASE` |
| Audio codec | AAC | — |
| Audio bitrate | 128k | `SEG_AUD_KBPS` |
| Sample rate | 44100 Hz | `SEG_AUD_RATE` |
| Channels | Stereo (2) | — |

These are defined as constants at the top of `useFFmpeg.js` and must be referenced by name, never hardcoded inline:

```js
const SEG_FPS       = 30
const SEG_TIMEBASE  = 90000
const SEG_PIX_FMT   = 'yuv420p'
const SEG_AUD_RATE  = 44100
const SEG_AUD_KBPS  = '128k'
```

## Filter Builders (`useFFmpeg.js`)

These pure functions are the only place FFmpeg filter strings are constructed. Never build filter strings inline inside the export pipeline.

| Function | Purpose |
|---|---|
| `buildEqFilter(b, c, s)` | Returns `eq=` filter string or `null` if all zero |
| `buildVideoPlacementVf(W, H, zoom, panX, panY)` | `-vf` string: fit → even → pad → zoom/pan crop. Safe for video demuxer only |

| `buildAtempoChain(speed)` | Returns `atempo` chain array for audio speed adjustment (handles speeds outside 0.5–2× range by chaining) |

## Transition Mapping

```js
const XFADE_MAP = {
  crossfade: 'fade',      slide_left: 'slideleft',
  slide_up:  'slideup',   zoom_in:    'zoomin',
  dip_black: 'fadeblack',
}
```

`transition === 'none'` means a hard cut with no overlap. It bypasses `XFADE_MAP` entirely:
- In Stage 2 Path A (all cuts, no music): concat demuxer stream-copy.
- In Stage 2 Path C (mixed): `concat` filter + optional timebase normalisation (see Stage 2 above).

`globalTransition` defaults to `'crossfade'` and can be set to any value including `'none'`. The Duration slider in Inspector is hidden when global is `'none'` (duration is irrelevant for hard cuts).

Per-clip `transition` overrides `globalTransition`. Setting it to `null` or `''` falls back to global. Setting it to `'none'` forces a cut on that specific clip regardless of global.

### Custom font upload

Users can upload `.ttf`, `.otf`, or `.woff` fonts via the Inspector. The font data is stored as a `Uint8Array` on the text segment (`customFontData`). In `Preview.jsx`, the `FontFace` API registers it as `MomCustomFont`. In export, the bytes are written to the WASM FS as `font_custom.ttf` using `.slice()`.

Custom fonts are persisted in saved workflows via the `customFonts` content-hash table. On workflow load, `customFontData` is re-attached to the segment with a fresh `.slice()` and `Preview.jsx` automatically re-registers it — no re-upload required.

## Error Surfacing

The `run()` helper wraps every `ff.exec()` call. On failure (non-zero return code or exception), it pushes the step label and the last 25 lines of FFmpeg log output to `pushLog` so the full error is visible in the LogDrawer, then throws to abort the pipeline.

The FFmpeg log listener is attached **once** in `load()` and **never removed**. Do not call `ff.off('log')` at any point.

## Image Effect Zoompan Notes

- FFmpeg.wasm 5.1 image2 demuxer with `-loop 1` delivers at 25 fps internally regardless of any `-r` flag on the input.
- The effect filter must use `fps=25` at its start to align the timebase before zoompan sees the stream.
- Zoompan expressions must use `on` (output frame index, 0-based) — not `in`, which evaluates to 0 every frame and produces a static image.
- Image effects are applied **within the single placement pass** as an additional filter chain node on `[vout]` → `[vfinal]` inside the `filter_complex`. The effect animates the already-composed (blur+placement) frame — not the raw source pixels. There is no intermediate file between placement and effect.

## Known Gaps / Future Work

- **`fileDuration` for direct timeline drops** — when files are dragged directly to the timeline (bypassing the media panel), `fileDuration` is `null` because no `<video>` element fires `onLoadedMetadata`. The TrimBar does not appear until the user previews the clip in the media panel first.
- **Speed + trim interaction** — `actualDur = (effectiveTrimEnd - trimStart) / speed`. This is correct for wall-clock output duration. The audio extraction mirrors this with identical trim/atempo args.
- **Music file not saved in workflow** — browser security prevents saving binary file references. The user must re-add the music file after loading a workflow. `musicTrimStart`, `musicTrimEnd`, and `musicVolume` are saved and restored correctly.
- **No undo/redo** — all mutations are direct `setState` calls.
- **Blur not rendered in Preview UI** — `blurBackground` is an export-only effect. The Preview component does not render the blur fill at runtime; it is only applied during FFmpeg export.
- **Image effects + blur together** — when both `blurBackground` and `imageEffect` are active, the blur fill is composited and the zoompan effect animates the composite — all within a single `filter_complex` pass.
- **Per-clip transition duration** — the `transitionDuration` field exists on clips but Stage 2 currently uses a single global `td` for all xfade/acrossfade calls. Per-clip override is stored but not yet applied in export.
- **Multiple custom fonts** — the current implementation supports one active custom font at a time (`font_custom.ttf`). If multiple text segments use different uploaded fonts simultaneously, only one will be written to the WASM FS. Each segment stores its own `customFontData`, so the data is preserved — the export pipeline would need to be extended to write per-segment font files with unique names.
- **Media re-matching is filename-only** — on workflow load, clips waiting for media are matched by filename alone. A file with the same name but different content will be silently accepted.
- **`-fps_mode cfr` replaces `-vsync cfr`** — FFmpeg 5.1 (the version in the WASM binary) deprecated `-vsync` in favour of `-fps_mode`. All normalise passes use `-fps_mode cfr`. Do not revert to `-vsync`.
- **Attached picture streams in video files** — some video files (particularly from mobile devices) embed an MJPEG thumbnail as a second video stream (`Stream #0:2: Video: mjpeg ... (attached pic)`). Always use `-map 0:v:0` (not `-map 0:v`) when mapping the video stream from a source file to avoid accidentally including the thumbnail in the output, which crashes the MP4 muxer.
